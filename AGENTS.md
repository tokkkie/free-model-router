# AI Agents Common Rules

> **重要**: このファイルはすべてのプロジェクトに共通して適用されるルールです。
> 各プロジェクト固有のルールは、そのプロジェクトの `AGENTS.md` を参照してください。

各プロジェクトに対する作業の際は、まずこの共通ルールを確認し、次に該当プロジェクトの `AGENTS.md` を確認してから行うこと。
プロジェクトについての詳細は `./README.md` も参照すること。

## 🚨 STOPPERS（即時停止・禁止事項）

以下は**絶対禁止**。例外は承認フロー経由のみ。

| 禁止事項 | 違反時の対応 |
|---------|------------|
| **無断の追加修正** | 「ついでに」の修正は承認なし禁止。必ず提案→承認 |
| **計画なき独走** | 大きな変更前に「最小差分の作業計画」を提示し承認を得よ |
| **エラー無視** | コマンド失敗（Exit != 0）は即停止。無視して続行を禁ずる |
| **完了報告のフライング** | 修正後は必ず `git add`/`git commit` 実行し成功まで報告禁止 |
| **破壊的変更の隠蔽** | API/DB/関数名変更時は「Breaking Change」と明示し事前承認 |
| **main直接操作** | `main`への直接Push厳禁。必ずトピックブランチ→PR形式 |
| **巨大なPR** | 目安3〜5ファイル以内（レビュー可能な粒度）。超過は分割または承認必須 |
| **範囲外の編集** | 指示対象ファイル以外の勝手な修正禁止。必要なら提案に留めよ |
| **仕様の推測実装** | 不明点を推測で補完して実装することを禁止。必ず確認または「不明」と報告 |
| **無断フォーマット変更** | prettier/go fmt等による全体変更は承認なし禁止 |
| **テスト未実行完了** | テスト/ビルド未確認での完了報告を禁止 |
| **仕様の意味変更** | 既存挙動の意味を変える変更は禁止（リファクタでも同様） |
| **自動ループの放置** | エラー解決が3回以内に完了しない場合、原因仮説と試行内容を整理して報告せよ |
| **環境基盤の無断変更** | `mise`設定、言語ランタイムのバージョン、パッケージマネージャ自体の更新、DB・S3・外部API等の状態変更を無断で行うことを禁ずる。影響範囲（他プロジェクト/CI）を伴う変更は禁止 |
| **バージョン管理外の重要ファイル操作** | `.git`ディレクトリ自体の編集、`.env`等の無断削除を禁止 |
| **競合の無視** | コンフリクト・他ブランチ変更を無視して上書きすることを禁止 |
| **証跡の破壊** | ログ・履歴・監査情報の削除/上書きを無断で行うことを禁止 |

### 巨大なPRの例外承認フロー
```
1. 理由を明示（「リファクタリングのため一括変更必要」等）
2. 影響範囲を明記
3. 分割不可の理由を提示
4. ユーザー承認後に実行
```

## 🛠️ TECHNICAL SPEC（技術仕様）

### BASE（共通固定）
- **Language**: 対話・コメントは**日本語を基本**。英語の長文解説を禁止
- **識別子命名**: スネークケース（`snake_case`）またはケバブケース（`kebab-case`）を基本。言語規約がある場合はそれに従う
- **Encoding**: UTF-8固定（BOMなし）。改行コードはLF。日本語コメントもUTF-8（BOMなし）とする
- **Path Handling**: スクリプト内での相対パス（`../` 等）は避け、プロジェクトルート起点のパス指定を優先せよ
- **Code Block Labels**: すべてのコードブロックには必ず言語名（`go`, `bash`, `diff` 等）を明記せよ
- **Reproducibility**: 同一手順で同一結果が得られること（環境差異に依存しない）
- **Time Handling**: タイムゾーンは明示（原則UTC）。ローカル依存禁止
- **Logging**: 構造化ログを優先。機密情報出力禁止
- **Dependency Management**: バージョン固定（lockfile必須）。依存追加時は既知の脆弱性を確認（CVE等）
- **Error Handling**: エラーは握り潰さず明示的に処理・ログ出力する

### OVERRIDE（プロジェクト固有で定義）
以下は各プロジェクトの `AGENTS.md` で **TECHNICAL SPEC (PROJECT OVERRIDE)** セクションに定義すること：

| 項目 | 定義内容 |
|------|----------|
| **Environment** | 実行環境（WSL/macOS/CI等）、ツール（mise等） |
| **Working Directory** | プロジェクトの絶対パス |
| **Retry** | リトライポリシー（必要な場合のみ） |
| **Testing** | テスト要件、カバレッジ基準 |
| **Language/Framework固有** | プロジェクト固有の技術制約（Go/GAE、Frontend等） |

## 🔐 SECURITY BASELINE（セキュリティ基本原則）

### 秘匿情報管理（絶対遵守）

| 対象 | 禁止事項 | 正しい方法 |
|------|---------|----------|
| APIキー | コード・ログ・コミット・チャットへの含め | 環境変数またはSecret Manager等 |
| パスワード | 同上 | 同上 |
| トークン | 同上 | 同上 |
| 設定ファイル（`*.yaml`, `*.tfvars`等） | チャットログ・コミットメッセージへの含め | 抽象表現で言及（「OAuth設定を更新」） |
| **.gitignore** | `.gitignore` 対象ファイルの強制的な `add/commit` | 尊重し無視対象ファイルはGit管理から除外 |
| **外部送信** | ユーザーの許可なき外部ツールへのコード・データ送信 | プロンプト内で完結させる |
| **ログ管理** | 本番ログに個人情報・機密情報を含める | ログ出力前にフィルタリング |
| **CI/CDシークレット** | CI設定ファイルにシークレットを直書き | 環境変数またはSecret Manager使用 |
| **URL/ドメインの秘匿** | 外部リソース参照時、URLに機密情報（トークン・プロジェクトID等）が含まれる場合は伏せ字にするか、ドメインのみを提示せよ |
| **コンテキストへの流し込み** | ユーザーが提供した秘匿情報（private-settings等）を別プロジェクトのコンテキストとして要約・転記することを禁止 |

### 権限管理
- **最小権限の原則**: 外部サービス/APIは最小権限で使用
- **権限昇格の禁止**: 不要な管理者権限（sudo等）の使用を禁止
- **外部送信の明示**: 外部ツールへ送信前に「送信対象・目的」を明示せよ
- **依存の脆弱性**: 依存追加時は既知の脆弱性を確認（CVE等）

### 公開範囲管理
- **プライベートリポジトリをPublicに変更することを絶対に禁止**
- 変更が必要な場合は事前承認必須（複数人の承認推奨）

### バックアップ／リカバリ意識
- 破壊的変更（DB操作、ファイル一括削除等）前はロールバック手段（バックアップ/スナップショット）を確認せよ

### セキュリティインシデント時の対応
```
1. 即座に対象ファイル/情報を秘匿化
2. 影響範囲を特定
3. ユーザーに即時報告
4. 修正計画を提示し承認を得る
```

## 💬 COMMUNICATION & HYGIENE（コミュニケーションと衛生管理）

### 基本姿勢
- **事実ベースの簡潔さ**: 冗長な挨拶・謝罪・ハルシネーションを排除。結論を先に述べよ
- **証跡の明示**: 実行コマンド・結果・差分は**省略せず提示せよ**
- **最小差分主義**: 出力は全文ではなく、修正箇所の `diff` 形式または最小範囲のコードブロックに限定せよ。**10行以上、または複数ファイルの変更は必ず `diff` または `patch` 形式**
- **変更理由の明示**: 「なぜ変更するか」を必ず説明せよ
- **ロールバック可能性**: 変更は元に戻せる単位で行え
- **コマンド透明性**: 実行コマンドは省略せず提示（コピー可能な形）
- **曖昧表現の排除**: 「たぶん」「おそらく」等の曖昧表現を避け、不確実な場合は断定せず「不明」または「要確認」と明示

### 作業開始前（必須）
以下を簡潔に提示し承認を得てから作業開始：
- **目的**: この作業で達成すること
- **変更対象**: 具体的なファイル・関数
- **影響範囲**: 変更が及ぶ範囲
- **実施手順**: 最小差分での手順
- **前提条件**: 使用する前提（OS、ツールバージョン、依存状態）を簡潔に記載

### 物理構造の確認（必須）
- 作業開始時、可能な場合は `ls` / `tree` 等で現在のディレクトリ・ファイルの実在を確認せよ（記憶に頼るな）

### README起点の思考（強制）
```
1. タスク開始時はまず README.md を読む
2. 設計思想とディレクトリ構造を完全に把握する
3. 不明な点は「不明」と回答する
4. 推測による構築を絶対に禁ずる
```

### 不明時の動作
- 仕様が不明な場合は作業を停止し確認を求める
- 仮実装・仮置き禁止
- 複数の解釈が可能な点は、作業を中断してユーザーに確認せよ

### 軽微変更の例外
以下は事前承認なしで実施可能（ただし事後報告必須）：
- 明確なtypo修正
- コメント修正（意味変更なし）
- 未使用import削除

### 提案モード
不確実性が高い場合：
- 実装せず「選択肢」として提示
- メリット/デメリットを簡潔に示す
- ユーザー判断を仰ぐ

### Git操作規則

#### Commit Logic
- **1コミット = 1目的**（単一責任原則）
- **形式**: `prefix: Simple English`（中学レベルの平易な英語）

#### Push前自律検証（必須）
```bash
# 必ず実行し確認せよ
git status
git diff
git log --oneline -10
```
**確認項目**:
- [ ] 意図しない変更が含まれていないか
- [ ] 履歴が汚染されていないか
- [ ] 秘匿情報が混入していないか

#### README同期
- 構成変更時は `README.md` のディレクトリツリーと解説を**即時更新せよ**

#### 末尾改行
- すべてのコードファイル末尾に**改行（LF）を入れよ**

### 依存追加ルール
外部依存（ライブラリ・パッケージ等）の追加時は以下を義務付ける：

| 項目 | 必須内容 |
|------|----------|
| **理由** | なぜその依存が必要か |
| **代替案** | 他に検討した選択肢と棄却理由 |
| **承認** | 上記を提示し承認を得ること |

### 完了条件（Definition of Done）
タスク完了報告前に以下を確認：
- [ ] ビルド成功
- [ ] テスト成功（該当する場合）
- [ ] 差分確認済み（`git status`/`git diff`）
- [ ] README更新済み（構成変更の場合）
- [ ] 意図しない変更が含まれていない
- [ ] 秘匿情報が混入していない
- [ ] 一時ファイル・デバッグコードを削除しクリーンな状態

### 冪等性の確保
可能な限り、同じコマンドを2回実行しても安全な（副作用のない）手順を提示せよ

## 📝 適用範囲と階層

```
ルート AGENTS.md（このファイル）
    ↑ 全プロジェクトに強制適用
    │
各プロジェクト AGENTS.md（差分定義）
    ↑ プロジェクト固有の上書き・追加
    │
実際の作業
```

### 優先順位（衝突時の適用順）
1. **ユーザーからの直接指示**（Prompt）
2. **プロジェクト固有ルール**（`./project/AGENTS.md`）
3. **共通ルール**（ルート `AGENTS.md`）

### 各プロジェクトAGENTS.mdの必須記述
各プロジェクトの `AGENTS.md` は冒頭に以下を必ず記載すること：

```markdown
> **重要**: このファイルはルート `AGENTS.md` を前提とする差分定義である。
> 共通ルール（STOPPERS・SECURITY BASELINE・COMMUNICATION & HYGIENE等）は必ずルートを参照すること。
